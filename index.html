1. Introducción

1.1 Propósito

El propósito de este Plan de Gestión de Configuración (PGC) es establecer los lineamientos, procesos, herramientas y responsabilidades necesarios para gestionar de manera adecuada los elementos de configuración del sistema EDI-301, incluyendo el control de versiones, la gestión de cambios, el seguimiento de artefactos y la administración de entornos.

Este documento asegura que todos los componentes del sistema sean identificados, controlados, auditados y mantenidos de manera consistente durante todo su ciclo de vida.

1.2 Alcance

Este plan aplica a todos los elementos relacionados con la construcción, despliegue y mantenimiento del sistema EDI-301, incluyendo:

Código fuente

  • Aplicación móvil en Flutter.
  • Backend en Node.js / Express.
  • Scripts SQL de base de datos.
  • Archivos de configuración, assets e imágenes.

Artefactos documentales

  • Documentación técnica (diseño, arquitectura).
  • Manuales (usuario, mantenimiento, pruebas).
  • Especificaciones y diagramas.
  • Planes de prueba y reportes de resultados.

Gestión de entornos

  • Entorno de Desarrollo (DEV).
  • Entorno de Pruebas (QA).
  • Entorno de Producción (PROD).

Fuera de alcance

  • Sistemas o servicios externos ajenos al proyecto.
  • Documentación institucional no relacionada directamente con el software.
  • Mantenimiento físico de infraestructura no controlada por el equipo EDI.

2. Roles y responsabilidades

Para garantizar un control adecuado de la configuración, se definen los siguientes roles y sus responsabilidades asociadas.

2.1 Administrador de configuración

  • Controlar los elementos de configuración (EC).
  • Mantener actualizada la CMDB.
  • Autorizar versiones oficiales y releases.
  • Coordinar auditorías de configuración.
  • Monitorear el cumplimiento del PGC.

2.2 Equipo de desarrollo (frontend / backend)

  • Registrar cambios en GitHub utilizando ramas establecidas.
  • Documentar modificaciones significativas en el código.
  • Crear versiones candidatas (Release Candidate).
  • Informar incidencias relacionadas con la configuración.

2.3 Equipo de QA

  • Validar la integridad de nuevas versiones en los entornos de prueba.
  • Verificar que los entornos reflejen la configuración aprobada.
  • Reportar defectos relacionados con versiones o inconsistencias de configuración.

2.4 Coordinador del proyecto

  • Autorizar cambios mayores en el sistema (cambios de alto impacto).
  • Aprobar versiones para su despliegue en producción.
  • Asegurar alineación con los requerimientos institucionales.

3. Control de configuración

El control de configuración garantiza que cualquier cambio realizado en el sistema sea autorizado, registrado, evaluado y documentado antes de integrarse definitivamente.

3.1 Identificación de elementos de configuración

Se consideran elementos de configuración (EC), entre otros:

  • Código fuente de la app móvil (Flutter).
  • Código fuente del backend (Node.js / Express).
  • Scripts SQL de creación y actualización de la base de datos.
  • Endpoints de la API documentados.
  • Manual de usuario, manual de mantenimiento, documentación de pruebas.
  • Archivos de recursos (imágenes, íconos, fuentes, multimedia).
  • Scripts de despliegue y archivos de configuración de entornos.

Cada elemento de configuración contará, como mínimo, con: identificador único, versión, tipo, ubicación en repositorio y responsable.

3.2 Control de versiones

Sistema de control de versiones: Git, alojado en GitHub.

Ramas principales

  • main: contiene la versión estable de producción.
  • dev: integra nuevas funcionalidades antes de estabilizar.
  • feature/<nombre>: desarrollo de funcionalidades específicas.
  • hotfix/<nombre>: correcciones urgentes en producción.
  • release/<versión>: preparación de una versión para despliegue.

Versionado semántico (SemVer)

Formato recomendado: MAJOR.MINOR.PATCH

  • MAJOR: cambios incompatibles.
  • MINOR: nuevas funcionalidades compatibles.
  • PATCH: correcciones de errores y ajustes menores.

Ejemplo de versión: v2.1.4

3.3 Control de cambios

El proceso de control de cambios se compone de las siguientes etapas:

  1. Registro de la solicitud de cambio (Issue en GitHub o formulario interno).
  2. Evaluación del impacto técnico y funcional.
  3. Aprobación del cambio por el Coordinador del proyecto o Administrador de configuración.
  4. Implementación del cambio en la rama correspondiente.
  5. Revisión por medio de Pull Request.
  6. Ejecución de pruebas por QA.
  7. Integración a rama principal (dev o main) según corresponda.
  8. Actualización de la CMDB y documentación afectada.

3.4 Control de entornos

Los entornos de desarrollo, prueba y producción se gestionan de manera diferenciada para asegurar estabilidad y trazabilidad de las versiones desplegadas.

Entorno Propósito Contenido típico
DEV (Desarrollo) Trabajo activo de los desarrolladores. Código en progreso, nuevas funcionalidades, pruebas iniciales.
QA (Pruebas) Validación por parte de QA y usuarios clave. Versión candidata, datos de prueba controlados, logs activos.
PROD (Producción) Uso real por parte de alumnos y colaboradores. Versión estable aprobada, datos reales, monitoreo permanente.

Los cambios en producción requieren autorización formal y registro específico en la CMDB y en el historial de despliegues.

4. Base de datos de gestión de configuración (CMDB)

La CMDB (Configuration Management Database) almacena el inventario de todos los elementos de configuración, sus versiones, estados y relaciones.

Cada entrada en la CMDB incluye, al menos:

  • Identificador del elemento de configuración.
  • Categoría (código, documento, script, recurso, etc.).
  • Versión actual.
  • Estado (en desarrollo, en pruebas, aprobado, en producción).
  • Ubicación (URL de repositorio, ruta interna, etc.).
  • Responsable técnico.

Ejemplo de registro en la CMDB

Elemento de configuración Versión Estado Ubicación Responsable
Backend API EDI-301 v1.5.2 Producción repo/backend-api Desarrollador backend
App móvil EDI-301 (Flutter) v2.0.1 QA repo/app-movil Desarrollador móvil
Scripts BD EDI-301 v1.3 Aprobado repo/db-scripts DBA / Backend

La CMDB puede implementarse como un archivo JSON, una hoja de cálculo compartida, una wiki interna o una sección específica dentro del repositorio de GitHub.

5. Seguimiento del estado de configuración

El seguimiento del estado de configuración (Configuration Status Accounting) consiste en registrar y reportar el estado actual de cada elemento de configuración a lo largo del tiempo.

Incluye:

  • Historial de versiones y cambios.
  • Estados de aprobación de cambios (aprobado, rechazado, pendiente).
  • Reportes de estado por rama (main, dev, release).
  • Registro de hotfixes y parches aplicados.
  • Registro de despliegues a QA y producción.

Ejemplo de reporte de estado de configuración

Elemento Versión actual Última modificación Responsable
App EDI-301 (móvil) v2.1.0 12/11/2025 Equipo móvil
API REST EDI-301 v1.9.3 10/11/2025 Equipo backend
Esquema BD EDI-301 v1.4 09/11/2025 DBA

6. Auditorías de configuración

Las auditorías de configuración garantizan que todos los artefactos del sistema estén correctamente registrados, actualizados y alineados con las versiones desplegadas en los diferentes entornos.

6.1 Auditoría funcional

La auditoría funcional verifica que:

  • Las versiones documentadas coincidan con las versiones implementadas.
  • Los cambios aprobados se hayan aplicado correctamente.
  • No existan componentes sin registrar en la CMDB.
  • La documentación técnica corresponda a la versión actual del sistema.

6.2 Auditoría física

La auditoría física confirma que:

  • Los archivos presentes en DEV, QA y PROD corresponden a las versiones aprobadas.
  • No haya archivos obsoletos, duplicados o no autorizados en el repositorio.
  • Las rutas y ubicaciones declaradas en la CMDB sean correctas.

6.3 Periodicidad

  • Auditoría menor: mensual, enfocada en cambios recientes.
  • Auditoría completa: trimestral, revisión integral de elementos y entornos.
  • Auditoría extraordinaria: posterior a incidentes críticos o fallos graves.
